# Multilingual Video Generation Pipeline Audit Report

This report documents the deep-dive audit of the OmniClip video generation pipeline when processing non-Latin scripts (Devanagari Hindi and Gujarati). It identifies current bugs, explains their technical root causes, and proposes step-by-step code solutions.

---

## 1. Pexels Stock Footage Irrelevance (Keyword Translation Failure)

### Detected Bug
When generating videos for Hindi or Gujarati topics, the pipeline fetches completely irrelevant stock video clips (e.g., temples, streets, monuments) instead of contextually relevant clips (e.g., technology, robots).

### Technical Root Cause ("The Why")
1. **Target-Language Query Transmission**: The Gemini AI model generates the `visual_keyword` in the target language (Hindi or Gujarati characters) matching the timeline text.
2. **Pexels Index Limitation**: The Pexels API video search index has poor localization and tagging support for Devanagari and Gujarati scripts.
3. **Query Mismatch Fallback**: Searching with keywords like `"रोबोट"` (robot) or `"વિજ્ઞાન"` (science) fails to return relevant matches. The search either falls back to generic Indian culture-related keywords (resulting in temples, street scenes, etc.) or fails completely, triggering cloned fallbacks of unrelated clips.

### Proposed Code Solution
Force the Gemini AI model to **always** generate the `visual_keyword` in English, even when the rest of the script (`voiceover_script` and timeline `text` segments) is generated in a regional language.

#### Steps to Implement:
1. Modify `apps/engine/services/gemini_service.py` Pydantic schema:
   ```python
   class TimelineSegment(BaseModel):
       chunk_index: int = Field(..., description="Sequential index of the segment starting at 0")
       text: str = Field(..., description="The exact words spoken in this segment (7 to 9 words)")
       visual_keyword: str = Field(..., description="A unique, context-specific keyword for video search. You MUST provide only 1 or 2 simple words for the visual keyword in English (e.g., 'robot', 'city', 'technology') even if the transcript/script is in Hindi or Gujarati. DO NOT write long sentences or descriptive phrases, as this breaks the Pexels API search.")
   ```
2. Update the `system_instruction` in [gemini_service.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/engine/services/gemini_service.py) under the rules section:
   ```python
   "RULES — you must follow all of these precisely:\n"
   "1. Split voiceover_script into sequential segments of 7 to 9 words each. "
   "   The 'text' field of each segment must be the exact words from that portion of voiceover_script.\n"
   "2. Every visual_keyword must be 100% unique across all segments — no keyword may be "
   "   reused or duplicated. You MUST provide only 1 or 2 simple words in English for the visual keyword (e.g., \"robot\", \"city\", \"technology\"). DO NOT write long sentences or descriptive phrases in Hindi/Gujarati, as this breaks the Pexels API search.\n"
   ```

## 2. Choppy and De-synchronized Subtitles (Word-Level Timing Fallback)

### Detected Bug
Subtitles appear jerky, freeze on screen, or lag behind the spoken voiceover, failing to align with the actual speed of speaking in Hindi and Gujarati.

### Technical Root Cause ("The Why")
1. **Discarded Whisper Timestamps**: [WhisperAlignmentService.align()](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/services/whisper_service.py#L254) loads the model and transcribes the audio using `word_timestamps=True`. It extracts the correct word-level timestamps from the voiceover. However, it discards the individual word timestamps, returning only the segment-level `start_time` and `end_time` mapped to the timeline.
2. **Uniform Division Fallback**: In [tasks.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/tasks.py#L168-L185), the pipeline reconstructs word-level timestamps by splitting the timeline text on whitespace and distributing the segment's total duration *uniformly* across all words:
   ```python
   word_dur = (seg_end - seg_start) / len(seg_words)
   ```
3. **Speech Cadence Mismatch**: Hindi and Gujarati sentence structures have different lengths and contain multiple short function words (e.g., "के", "में", "हो", "છે", "થી") that are spoken in milliseconds. Uniform division assigns the same duration to all words, creating a massive visual sync lag and making the text transition feel choppy.

### Proposed Code Solution
Pass the actual high-precision word timestamps generated by Whisper directly down to the subtitle generator.

#### Steps to Implement:
1. Modify `WhisperAlignmentService.align()` in [whisper_service.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/services/whisper_service.py) to return a tuple containing the aligned segment timeline and the raw `words` list:
   ```diff
   -    def align(cls, audio_path: str, timeline: list) -> list:
   +    def align(cls, audio_path: str, timeline: list) -> tuple[list, list]:
   ```
2. Update the fallback path in `align()` to construct dummy uniform word lists if Whisper fails:
   ```python
   # Inside _uniform_fallback, return (enriched, dummy_words)
   ```
3. Update [tasks.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/tasks.py) to receive both values:
   ```python
   aligned_timeline, real_words = WhisperAlignmentService.align(audio_path, raw_timeline)
   ```
4. Feed `real_words` directly to `sentence_aware_chunk(real_words)` to preserve genuine spoken timestamps instead of executing the artificial uniform division block.

## 3. FFmpeg Subtitle Rendering Glitches (Matra/Ligature Detachment and Overlaps)

### Detected Bug
Complex scripts (Devanagari, Gujarati) suffer from font rendering glitches in FFmpeg where vowel signs ("matras") detach from their base consonants, letters overlap, or text clip areas overlap on short duration intervals.

### Technical Root Cause ("The Why")
1. **Dynamic Font Scaling Conflict**: The ASS style generator in [ffmpeg_service.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/services/ffmpeg_service.py#L154) applies a scale-in transition:
   ```python
   bounce_tag = f"{{\\fscx110\\fscy110\\t(0,{_BOUNCE_ENTRY_MS},\\fscx100\\fscy100)}}"
   ```
   For short-duration words (e.g., 0.1s to 0.3s), the shaper library (`HarfBuzz` inside FFmpeg `libass`) must recalculate complex glyph combinations for Devanagari/Gujarati ligatures at a rapidly changing font scale on every single frame. This dynamic rendering is computationally expensive and causes rounding errors in glyph margins, leading to overlaps and matra detachment.
2. **Font Family Incompatibility**: The subtitle styling utilizes `"Arial Black"`:
   ```python
   _ASS_FONT_NAME = "Arial Black"
   ```
   Arial Black does not contain native Devanagari or Gujarati glyph tables. FFmpeg must perform font substitution. The fallback system fonts (e.g., MS Gothic, Mangal) have different horizontal/vertical metrics, creating alignment glitches when outline and shadow layers are composited.

### Proposed Code Solution
1. Use a language-appropriate font family (such as `"Nirmala UI"` or `"Arial Unicode MS"`) when non-Latin text is detected, to ensure layout metrics match perfectly across layers.
2. Disable the dynamic scaling bounce tag (`bounce_tag`) for complex script characters, or bypass it entirely if subtitle duration is short.
3. Keep the minimum floor duration (`_MIN_DURATION_S`) at a higher value (at least `0.4s`) to allow the shaper engine to settle.

#### Steps to Implement:
1. Add a language detection helper to check if text contains Devanagari/Gujarati characters:
   ```python
   def _contains_complex_script(text: str) -> bool:
       return any(ord(char) in range(0x0900, 0x098F) or ord(char) in range(0x0A80, 0x0AFF) for char in text)
   ```
2. Dynamically swap `Fontname` in `_generate_ass_subtitles`:
   ```python
   font_name = "Nirmala UI" if _contains_complex_script(display_text) else "Arial Black"
   ```
3. Disable the bounce tag for complex scripts:
   ```python
   bounce_tag = "" if _contains_complex_script(display_text) else f"{{\\fscx110\\fscy110\\t(0,{_BOUNCE_ENTRY_MS},\\fscx100\\fscy100)}}"
   ```

## 4. TTS EdgeTTS Fallback Failures (Voice Mismatches)

### Detected Bug
When the ElevenLabs API is down or the quota is exhausted, the system falls back to EdgeTTS. However, Hindi or Gujarati scripts are spoken with an English neural voice, causing silent audio or dropped words.

### Technical Root Cause ("The Why")
1. **Fallback Voice Discrepancy**: If `voice_id` is an ElevenLabs-specific voice ID (e.g., `JBFqnCBsd6RMkjVDRZzb`) or not registered in the `VOICE_MAP` keys:
   ```python
   _, edge_voice = VOICE_MAP.get(voice_id, (_DEFAULT_ELEVENLABS_VOICE, _DEFAULT_EDGE_VOICE))
   ```
   It falls back to the default `_DEFAULT_EDGE_VOICE` which is `"en-US-ChristopherNeural"`.
2. **Speech Model Failure**: Sending Hindi or Gujarati characters to an English-only text-to-speech voice causes EdgeTTS to bypass the text or output silence, leading to corrupt audio files.

### Proposed Code Solution
Incorporate script text inspection to dynamically select the EdgeTTS voice if the requested voice lookup fails or falls back to an incompatible default language.

#### Steps to Implement:
1. Define native language fallbacks in [audio_service.py](file:///c:/Users/DELL/Desktop/OmniClip/apps/processor/services/audio_service.py):
   ```python
   _DEFAULT_HINDI_EDGE_VOICE = "hi-IN-MadhurNeural"
   _DEFAULT_GUJARATI_EDGE_VOICE = "gu-IN-NiranjanNeural"
   ```
2. Refactor `EdgeTTSProvider.generate` voice lookup logic to audit the script text:
   ```python
   # Detect text language
   has_hindi = any(ord(c) in range(0x0900, 0x097F) for c in text)
   has_gujarati = any(ord(c) in range(0x0A80, 0x0AFF) for c in text)
   
   # Map voice
   mapping = VOICE_MAP.get(voice_id)
   if mapping:
       edge_voice = mapping[1]
   else:
       # Fallback based on character sets
       if has_hindi:
           edge_voice = _DEFAULT_HINDI_EDGE_VOICE
       elif has_gujarati:
           edge_voice = _DEFAULT_GUJARATI_EDGE_VOICE
       else:
           edge_voice = _DEFAULT_EDGE_VOICE
   ```
